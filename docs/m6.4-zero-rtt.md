## 1. What TLS 1.3 0-RTT does in QUIC

When a TLS 1.3 client has a session ticket from a prior handshake
with the same server, it can send application data *alongside* its
ClientHello — the "early data" window. The server processes that
data before the handshake completes, saving a round-trip on
reconnect.

`rustls` controls this via `ServerConfig::max_early_data_size`
(bytes the server will buffer for early data per resumed connection).
Default in rustls 0.23: `0`, i.e. 0-RTT off. To opt in, set it to
a non-zero value.

`quinn` 0.11 + `rustls` 0.23 propagate this setting transparently
through `quinn::crypto::rustls::QuicServerConfig::try_from(...)`.

## 2. What v0.4 turns on

```rust
// proteus-core/src/tls.rs
pub const MAX_EARLY_DATA_BYTES: u32 = 16 * 1024;

// in server_config():
rustls_cfg.max_early_data_size = MAX_EARLY_DATA_BYTES;
```

16 KiB is comfortably above the size of a single PROTEUS AUTH_REQUEST
(≈ 130 bytes: 16-byte frame header + 1-byte cid_len + ≤ 64-byte
client_id + 32-byte nonce + 64-byte Ed25519 signature) and leaves
headroom for future v0.4+ control-stream chatter without operators
having to retune.

Client-side: no config change required. rustls defaults already
enable session ticket caching for clients. When a PROTEUS client
reconnects to a server it's recently talked to, it'll automatically
present its cached ticket and attempt 0-RTT.

## 3. Why this is replay-safe for PROTEUS

0-RTT data is replayable: an attacker who recorded the
ClientHello + early-data bytes on the wire can resend them later,
and the server has no way to distinguish replay from a fresh
connection until the rest of the handshake completes.

For PROTEUS specifically, the *only* data a client sends in the 0-RTT
window is the AUTH_REQUEST frame on the freshly-opened control bidi.
PROTEUS catches replayed AUTH_REQUESTs the same way regardless of
0-RTT vs 1-RTT delivery:

1. **Per-client `(client_id, nonce)` replay cache** (M7,
   `proteus_core::replay::ReplayCache`). 300-second TTL by default.
   A replayed AUTH_REQUEST has the same nonce → cache hit → server
   responds with `H3_GENERAL_PROTOCOL_ERROR` close. Same code path
   as for a 1-RTT replay.

2. **TLS exporter binding** (M5 / M6). The AUTH_REQUEST signature
   covers `"PROTEUS-v0.3-auth" || exporter || nonce`. The exporter
   is per-TLS-session, so even a replay within the cache TTL on a
   *different* TLS session fails signature verification.

3. **No PROTEUS state-changing side effect happens until after
   AUTH_RESPONSE is sent.** Proxy streams open only after the client
   reads the AUTH_RESPONSE — and AUTH_RESPONSE is sent only over
   1-RTT keys (because the server hasn't finished the handshake at
   the moment it processes the 0-RTT AUTH_REQUEST; the response goes
   out after handshake completion). So even if 0-RTT data is
   replayed and momentarily accepted, the *side effects* (proxy
   target connect, byte forwarding) wait for 1-RTT confirmation.

Combined, these three mean PROTEUS-on-0-RTT has the same effective
replay-safety as PROTEUS-on-1-RTT: every replayed AUTH_REQUEST
fails at one of (signature mismatch, cache hit) and the connection
closes generically.

The QUIC layer additionally has its own anti-replay machinery
(rustls's session-ticket window), but PROTEUS does not depend on
it — the cache + exporter binding above are sufficient on their
own.

## 4. What's NOT in this commit

* **Client-side 0-RTT trigger.** Quinn exposes
  `Connecting::into_0rtt()` for clients that want to send data with
  their ClientHello. The v0.4 SOCKS5 client is currently long-lived
  — it dials once at startup and keeps the connection — so 0-RTT
  has zero practical benefit for the current proteus-client binary.
  Future short-lived clients (per-process, frequent reconnects)
  would call `into_0rtt()`; that's not in this commit.

* **Persistent session-ticket storage.** rustls's default
  `ClientSessionMemoryCache` is in-process; tickets die when the
  client process restarts. Persistent storage is a future operator
  feature.

* **Live integration test.** Reaches into M9.4 (rc.1 sign-off).
  Test needs:
  1. Refactor `proteus-server`'s auth-policy-proxy path to be
     callable as a library function rather than only from `main()`
     (also required for M7.4 migration test).
  2. Spin up server in-process, do a fresh 1-RTT handshake +
     record session ticket.
  3. Tear down client. Re-create client with the same in-memory
     ticket store.
  4. Reconnect — verify `Connecting::into_0rtt()` returns Ok and
     `accepted` future resolves to `true`.
  5. Send an AUTH_REQUEST as early data, observe server-side
     processing.
