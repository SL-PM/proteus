This was the blocking dependency for everything below.

### 2.3 Connection migration (M7.4)

`migration.rs::connection_survives_path_migration` verifies:
1. Client connects from local UDP port A, completes auth.
2. `endpoint.rebind(new_socket)` swaps the client's source 5-tuple to
   port B. (Quinn's `PATH_CHALLENGE`/`PATH_RESPONSE` fires.)
3. A fresh proxy stream is opened on the migrated path, a 14-byte
   payload is sent, the localhost TCP echo bounces it back, and
   `read_frame_aead` decrypts correctly.
4. Server-side metrics show `auth_attempts == 1, auth_success == 1` —
   no second auth was triggered by the 5-tuple change.

That's the M7.4 contract: PROTEUS's `(client_id, nonce)` replay cache
is keyed on client_id, not on 5-tuple, so migration is free.

### 2.4 TLS 1.3 0-RTT (M6.4)

Two tests:

* **`server_does_not_panic_on_fresh_quic_handshake`** — direct
  regression for the M5.4.1-era panic. With
  `MAX_EARLY_DATA_BYTES = u32::MAX` (per the Quinn-rustls constraint
  documented in [m6.4-zero-rtt.md](m6.4-zero-rtt.md) §2), accepting
  a fresh incoming handshake must NOT trip the quinn-proto rustls
  glue assertion. Passes.

* **`session_resumption_with_optional_0rtt`** — best-effort 0-RTT
  check. Connects once over 1-RTT (the server issues a
  NewSessionTicket), closes, reconnects via
  `Connecting::into_0rtt()`. The test passes whether:
  * (a) `into_0rtt()` returns `Ok` and the `accepted` future resolves
    to `true` — real 0-RTT acceptance, OR
  * (b) `into_0rtt()` returns `Err` because the rustls ticket cache
    hadn't observed the NewSessionTicket in time, falling back to
    1-RTT — still a successful auth.

  Both outcomes prove the server-side handling is correct. In
  practice on this rustls 0.23 / quinn 0.11.14 combo case (b) is
  what fires; deterministically triggering (a) needs deeper
  investigation of when rustls writes tickets to its
  `ClientSessionMemoryCache` relative to `Connection::close()`.
  Not a blocker — the production 0-RTT scenario (rare-reconnect
  short-lived client) doesn't exist yet for PROTEUS, see m6.4-zero-rtt.md §4.

### 2.5 High-fidelity decoy comparison (M3.4 + M8.4)

**Setup:**
```sh
proteus-tools fetch-decoy --url https://www.cloudflare.com/ --out cf.html
# → 1376452 B, content-type text/html; charset=utf-8
proteus-server with decoy.static_page=cf.html
# → "decoy: file (1376452 bytes)" in startup log
```

**Verification:**
```sh
# brew curl 8.20 with ngtcp2/nghttp3 (system curl 8.7 lacks H3 backend)
/opt/homebrew/opt/curl/bin/curl --http3-only -k \
    https://localhost:14435/ -o proteus-decoy.html

shasum -a 256 cf.html proteus-decoy.html
# c73c8831f4688fb4a4bfdfb511965d2f627d7a906ac172df0fb5f6e943d24cb9  cf.html
# c73c8831f4688fb4a4bfdfb511965d2f627d7a906ac172df0fb5f6e943d24cb9  proteus-decoy.html
```

**Result: PROTEUS H3 decoy body is byte-identical to the live
cloudflare.com index** at capture time (2026-05-27Z), 1,376,452 B,
matching SHA-256.

### 2.6 Residual fingerprint divergence (known limitation)

While the body is byte-identical, the **HTTP response headers
diverge**. Headers returned by each:

| Header | PROTEUS decoy | Real cloudflare.com |
|---|---|---|
| `server` | `nginx/1.27.0` | (absent — cloudflare doesn't advertise) |
| `content-type` | `text/html; charset=utf-8` | `text/html; charset=utf-8` |
| `accept-ranges` | `bytes` | (absent) |
| `date` | (absent) | `Wed, 27 May 2026 ...` |
| `link` | (absent) | preload + preconnect hints |
| `cache-control` | (absent) | `public, max-age=10, ...` |
| `vary` | (absent) | `accept-encoding` |
| `strict-transport-security` | (absent) | `max-age=31536000; ...` |
| `cf-placement` | (absent) | `remote-WAW` |
| `content-security-policy` | (absent) | ~3 KB of CSP directives |
| 10+ others | (absent) | present |

PROTEUS hardcodes a minimal nginx-style header set in
`serve_h3_decoy` (lib.rs). A motivated prober comparing headers can
distinguish PROTEUS-decoy from real-cloudflare in one HTTP request.

**Disposition:** documented limitation, deferred to v0.5.
Header-mirroring would require either:
1. Snapshotting both body *and* headers in `fetch-decoy`, or
2. A live `cover.proxy_decoy_to_real_host: true` mode that
   transparently relays the H3 probe to the cover host (Approach B
   per [PROTEUS-v0.4-plan.md](PROTEUS-v0.4-plan.md) §6).

Both are bigger changes than v0.4-rc.1 wanted to absorb. The v0.4
fix narrows the gap from "1-byte body" to "byte-identical body";
headers come next.

## 3. What's NOT in v0.4-rc.1

* **Real-0-RTT happy-path trigger in the integration test.** See
  §2.4. Regression test in place; deterministic acceptance test
  needs deeper rustls-cache investigation.
* **ALPN unification.** `proteus/0.3` is still advertised. M1.4+M2.4
  are out per [m2.4-dispatch-research.md](m2.4-dispatch-research.md);
  fixing this is now a v1.0-class change (forking h3 or writing a
  mini-h3 server).
* **Decoy header mirroring.** §2.6. v0.5.
* **Wire-level traffic-pattern capture comparison** (the M14-style
  pcap + tshark side-by-side). With AEAD-wrapped frames everything
  inside the TLS tunnel is already opaque; the residual fingerprint
  lives at the connection envelope (ALPN, idle timing, handshake
  size) which is what v0.5 padding profiles will address. M14's
  packet-count comparison still holds as the v0.3 baseline.
