# M12.5 — fingerprint-measurement sign-off

**Branch:** `v0.5/m11.5-fingerprint-eval`
**Roadmap:** measurement harness for the v0.5 wire-shaping work
**Status:** Done. v0.5 bucket-padding's effect is now **measured**, not
assumed — with an honest statement of exactly what it does and doesn't
buy.

---

## 1. What this measures

The well-posed question for *bucket* padding is **not** "does PROTEUS
look like real H3?" (that needs profile-matching we don't do + a
capture corpus we don't have). It is:

> Can a passive observer tell two different PROTEUS *activities* apart
> purely from server→client frame sizes?

That is exactly the leak padding addresses. The harness
(`crates/proteus-server/tests/fingerprint.rs`) drives real in-process
flows, records the raw on-wire `payload_len` of server→client frames,
and feeds them to the pure `proteus_core::fingerprint` math:
total-variation distance → best-possible classifier accuracy
`(1 + TV) / 2`.

* accuracy `0.5` = coin flip = **indistinguishable**
* accuracy `1.0` = always right = **trivially distinguishable**

## 2. Measured result

Three workloads (fixed echo payload sizes, 12 round-trips each),
observed wire sizes and the resulting best-classifier accuracy:

| comparison | padding | observed wire sizes | TV | best acc |
|---|---|---|---:|---:|
| small (5B) vs mid (100B) | **off** | 21 vs 116 | 1.000 | **1.000** |
| small (5B) vs mid (100B) | **on** | 128 vs 128 | 0.000 | **0.500** |
| small (5B) vs large (1000B) | **on** | 128 vs 1024 | 1.000 | **1.000** |

Reproduce:
```sh
cargo test -p proteus-server --test fingerprint -- --nocapture
```

## 3. Interpretation

* **Padding works on the fine-grained leak it targets.** Without
  padding, a 5-byte and a 100-byte response are perfectly separable on
  the wire (acc 1.000). With padding both become a 128-byte frame —
  the best any classifier can do collapses to a **coin flip (0.500)**.
  The per-frame size leak *within a bucket* is genuinely closed, and
  now we have the number to prove it rather than asserting it.
* **Padding is a quantizer, not a uniformizer (honest limit).** Across
  bucket boundaries the difference survives: a 5-byte (→128) and a
  1000-byte (→1024) activity stay perfectly distinguishable (acc
  1.000). Bucket-padding hides *how big within a bucket*, not *which
  bucket*. An operator wanting to hide coarse activity shape needs
  either coarser buckets (fewer, larger — more overhead) or
  profile-driven sampling.

## 4. Timing axis (M13.5)

The size measurement above used the live wire (deterministic). Live
inter-arrival timing is wall-clock-noisy, so the timing measurement
instead exercises the jitter *mechanism* deterministically: model a
perfectly regular send schedule (a strong fingerprint — e.g. a periodic
heartbeat at fixed cadence `G`) and apply the real `Jitter` sampler
with a seeded RNG, observing the gap `G + d_i − d_{i-1}` per frame
(`tests/fingerprint.rs::jitter_destroys_a_regular_timing_signature`).

| metric | no jitter | jitter [0,8] ms |
|---|---:|---:|
| distinct gap bins | **1** (spike at 10 ms) | **17** (spread over 2–18 ms) |
| mean cadence | 10.00 ms | 10.00 ms |
| TV(off, on) | — | **0.887** |

Interpretation:

* **Jitter destroys timing *regularity*.** A precise periodic cadence
  is a sharp single-bin signature; jitter spreads it across 17 bins
  (TV 0.887). A fingerprint that keys on "frames arrive every exactly
  10 ms" no longer fires.
* **Honest limit — jitter does NOT hide the *mean* cadence.** The
  average inter-arrival is preserved (10.00 → 10.00), because uniform
  jitter has zero mean effect on the gap. An analyst measuring *average*
  cadence still recovers `G`. Uniform jitter is a regularity-breaker,
  not a cadence-matcher — the timing analog of "padding is a quantizer,
  not a uniformizer." Matching a target cadence needs profile-driven
  timing.

## 5. What this does NOT measure (deferred)

* **PROTEUS vs. real H3.** Needs a captured/synthetic cover
  distribution and profile-matching. This harness measures
  *self-distinguishability* (activity A vs. activity B), which is the
  leak the shaping actually addresses — not mimicry.
* **Live end-to-end timing.** The timing axis models the jitter
  mechanism (the sampler that runs in production) without network
  noise. A pcap-level inter-arrival capture would add RTT/scheduler
  effects; `scripts/capture-*.sh` can produce that manually.
* **A real adversary's classifier.** TV gives the *theoretical optimum*
  for any classifier under equal priors; a real DPI box may do worse.
  We report the upper bound, which is the conservative (pro-defender-
  pessimistic) choice.

## 6. Why this milestone mattered

Before M11.5–M13.5 we had shipped three wire-shaping mechanisms
(padding, jitter, pacer) all carrying the same honest caveat: "we think
this helps." This harness turns those claims into measured numbers on
**both axes**:

* **Size (padding):** `1.000 → 0.500` within a bucket; `1.000` across
  buckets (the limit).
* **Timing (jitter):** regularity destroyed (1 → 17 bins, TV 0.887);
  mean cadence preserved (the limit).

Just as importantly, each measurement also pins the *limit*, so we
don't overclaim. The `fingerprint` module is now the foundation for
evaluating any future mechanism — most importantly profile-driven
sampling, which is the only thing that addresses the two documented
limits (across-bucket size + mean cadence).

### Profile-sampling feasibility (M14.5)

So we used the harness to ask whether profile-driven sampling is worth
building, *before* committing to the big wire integration. Against a
synthetic smooth cover size histogram
(`tests/fingerprint.rs::profile_sampling_beats_bucketing_against_a_cover`):

| approach | TV to cover | distinct bins |
|---|---:|---:|
| bucket-padding (round up to {128,256,512,1024,1500}) | **1.000** | 5 |
| profile-sampling (`Distribution::sample`, inverse-CDF) | **0.005** | 9 |

Bucketing scores TV 1.000 here because none of the cover's sizes
coincide with a bucket value, so the quantized 5-spike output is
*disjoint* from the cover — maximally distinguishable. (The exact 1.000
is partly an artifact of this cover having no on-bucket sizes; a real
cover would overlap somewhat. The qualitative result — sampling ≫
bucketing — is robust.) Drawing target sizes straight from the cover
histogram reproduces it almost exactly (TV 0.005).

**Conclusion:** profile-driven sampling clearly beats bucketing at
matching a cover, justifying the (deferred, data-dependent) full
integration. What that integration still needs: a real capture corpus /
profile loader, and a send path that picks a sampled target size then
pads/**fragments** to it — fragmentation being the hard part, since you
can pad a frame up but cannot shrink it.
