# M7.4 — Connection Migration (status note)

**Date:** 2026-05-27
**Roadmap:** [`PROTEUS-v0.4-plan.md`](PROTEUS-v0.4-plan.md) §M7.4
**Status:** Satisfied by Quinn defaults + PROTEUS's replay-cache design.
No PROTEUS code change required.

---

## What QUIC connection migration is

A QUIC connection is identified by its connection ID, not its 5-tuple
(src_ip, src_port, dst_ip, dst_port). A client can change any of those
mid-session — e.g. a phone switching from Wi-Fi to LTE — and the server
validates the new path via `PATH_CHALLENGE` / `PATH_RESPONSE` frames.
The session continues with the same keys, the same streams, the same
application state. RFC 9000 §9 spells out the details.

## Why PROTEUS gets this for free

1. **Quinn enables migration by default.** The default
   [`quinn::TransportConfig`] does not disable it.
   [`proteus_core::tls::default_transport_config`] only customises
   `max_idle_timeout = 30s` and leaves migration on.

2. **Replay cache is keyed on `client_id`, not 5-tuple.** See
   [`proteus_core::replay::ReplayCache::check_and_record`]. The
   `(client_id, nonce)` key survives a migration — the new path
   doesn't look like a new client to the cache.

3. **Per-target proxy streams are stream-scoped, not connection-
   scoped.** Stream IDs are independent of the underlying transport,
   so `proteus_core::proxy::bridge_quic_tcp` and `bridge_quic_udp`
   never see the change.

4. **The SOCKS5 frontend doesn't expose the QUIC peer's port.** A
   migration on the QUIC side has no visible effect on the local
   SOCKS5 listener.

## How a v0.4 client triggers migration

Client code calls [`quinn::Endpoint::rebind`] with a new
`UdpSocket`. Existing connections continue using the new socket; a
fresh `PATH_CHALLENGE` round trip is folded into the next packet.

The PROTEUS client binary (M9 SOCKS5 daemon) doesn't trigger
migration on its own — the operator or a higher-level orchestrator
calls into it. That's a v0.5-or-later operational concern; v0.4
just makes sure the protocol stack is migration-transparent.

## What verification looks like

The acceptance-criteria item for M9.4 sign-off (v0.4-rc.1) includes a
live integration test:

> 8. Connection migration integration test passes (client switches
>    source port mid-flight; session continues).

Implementation sketch for that test (lands as part of M9.4):

```rust
// tests/connection_migration.rs  (M9.4)
#[tokio::test]
async fn client_rebind_mid_session_preserves_proteus_session() {
    // 1. Spawn server in-process via a refactored
    //    proteus_server::serve_one(cfg) entry point.
    // 2. Client: bind UDP A, connect, run PROTEUS auth, open proxy
    //    stream, send 1 DATA frame.
    // 3. Client: endpoint.rebind(udp_socket_B)?
    // 4. Client: send another DATA frame on the existing stream.
    // 5. Server: receive it, echo back via the proxy.
    // 6. Assert payload roundtrip.
}
```

Not landed yet because the harness needs `proteus-server`'s auth +
policy + replay path callable as a library function, not just from
its `main()`. Refactoring that out is M9.4's prerequisite (and a
small, contained refactor — most of `proteus-server::main.rs` is
already pure async functions on `proteus-core` types).

## What does NOT change in v0.4

* **No new config field.** Migration is not opt-in.
* **No client-API change.** PROTEUS does not need a custom
  handshake for migration; the underlying `quinn::Endpoint::rebind`
  is the only API the operator needs.
* **No threat-model change.** Migration does not affect any of
  A1–A11. It marginally helps A8 (IP-block resilience on the
  *client* side — a client can move IP without losing session) but
  A8 is still out-of-scope for v0.4 because the *server's* IP
  doesn't move.
