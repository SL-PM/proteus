# M8.5 — v0.5-rc.2 sign-off (timing jitter)

**Branch:** `v0.5/m6.5-timing-jitter`
**Roadmap:** [`PROTEUS-v0.5-plan.md`](PROTEUS-v0.5-plan.md) §11 + §6 acceptance criteria
**Status:** Done. Sender-side inter-arrival jitter shipped and verified
end-to-end. Profile-driven (distribution-matching) timing remains
deferred — this is a *decorrelator, not a mimic*.

---

## 1. What v0.5-rc.2 ships

A **bounded random delay before each outgoing proxy-stream DATA frame**,
sampled uniformly from `[min_ms, max_ms]`. Attacks the *timing* axis of
A7 that rc.1 left untouched: before this, PROTEUS wrote each frame the
instant the bridge produced it, so inter-arrival gaps mirrored the
application's data-production timing — a deterministic signature.

Defining property: **sender-side only.** No wire-format change, no flag
bit, no lockstep upgrade. Each end enables jitter independently. (rc.1
bucket-padding, by contrast, needed the `FLAG_PADDED` bit + both ends in
agreement.)

Default OFF — a v0.5-rc.1 / v0.4 deployment is unchanged until the
operator sets `timing_jitter.enabled: true`.

- [x] **`TimingJitterConfig`, default off, validated.** `validate()`

- [x] **Send path applies the delay when enabled; disabled = no

      (bind → handle_proxy_stream → proxy_to_{tcp,udp} → bridge) and
      client-side (main → handle_socks5 → bridge). (M7.5)
- [x] **Integration test: data round-trips with jitter on.**
      `tests/jitter.rs` — single-frame echo intact + multi-frame
      ordering/AEAD-counter sync intact. (M8.5)
- [x] **Sign-off doc** (this file). (M8.5)
- [x] **No regression with jitter off.** All 151 pre-rc.2 tests pass
      unchanged; the disabled path is byte- and timing-identical to

      constructed).

## 3. Evidence

`tests/jitter.rs`, server started via `start_jittered("alice", …)`:

### 3.1 Correctness under jitter

`data_round_trips_with_jitter_enabled` — server jitter set to a
**constant** 20 ms (`min == max == 20`). A 1000-byte payload sent
through the proxy is echoed back **byte-identical**. This is the
load-bearing assertion: the `tokio::time::sleep` injected before each
DATA send does not corrupt the stream, desync the per-stream
ChaCha20-Poly1305 nonce counters, or drop bytes.

### 3.2 Jitter is actually on the path

Same test: the server→client echo frame cannot arrive faster than the
constant 20 ms delay. Measured `elapsed >= 15 ms` (15 ms lower bound to
absorb timer granularity; machine scheduling only ever *adds* time, so
this is not flaky). Confirms the delay is wired in, not a no-op.

### 3.3 Multi-frame integrity

`larger_transfer_survives_jitter` — 5 distinct frames sent back-to-back
with a `[0, 5] ms` random range; the reassembled echo byte-stream
equals the concatenation of what was sent, in order. Proves ordering +
AEAD-counter sync hold across many jittered sends.

## 4. What v0.5-rc.2 does NOT achieve (honest limitations)

* **Throughput cost.** Per-frame delay is not free. A uniform
  `[min, max]` delay adds an average of `(min+max)/2` ms of latency
  *per frame*; on bulk transfer this caps throughput at roughly
  `frame_size / avg_delay` (≈1.5 MB/s at 1 ms avg, ≈300 KB/s at 5 ms,
  ≈75 KB/s at 20 ms). The default range is small (`0–5 ms`) and the
  config docs carry this warning. A token-bucket pacer that jitters
  frame *boundaries* without serializing every write would cost less —
  future work.
* **Uniform ≠ mimic.** A real cover host's inter-arrival distribution
  is not uniform. rc.2 jitter *decorrelates* PROTEUS's send timing from
  the application's; it does not *match* any specific cover host. A
  statistical attacker (A7) who models the inter-arrival histogram can
  still distinguish uniform jitter from real H3 timing. Closing that
  needs profile-driven inter-arrival sampling — same recorded-corpus
  dependency as size sampling, deferred (see §6).
* **Sender-side only, by design.** Jitter shapes *our* send timing. It
  does nothing about RTT, congestion-control timing, or the receiver's
  ACK pattern.
* **No pcap-level cross-check in CI.** Evidence is in-process
  (frame-arrival timing at the QUIC stream layer). A tcpdump capture
  would show UDP-packet timing one layer out; `scripts/capture-*.sh`
  can produce that manually (needs root), not wired into the suite.

**Net:** rc.2 breaks the *deterministic* send-timing signature (A7
timing axis, partial) at a tunable throughput cost. It does not claim
to defeat a distribution-modelling analyst — that, plus the size-
distribution half, is the bar for profile-driven sampling.

## 5. Threat-model delta

| Adversary | v0.5-rc.1 | v0.5-rc.2 |
|---|---|---|
| A5 active DPI / wire-pattern | mostly closed (size quantized, idle filled) | unchanged |
| A7 statistical analysis | partial (coarse size quantization only) | **partial+** — send-timing decorrelated; distribution shape (size *and* timing) still differs |
| A6, A8–A11 | unchanged | unchanged |

## 6. Deferred to a later increment

* Profile-driven **size** sampling (rc.1 leftover) and profile-driven
  **inter-arrival** sampling — both consume
  [`fingerprint-profile.example.yaml`](fingerprint-profile.example.yaml)
  and need a capture corpus or synthetic model.
* Token-bucket pacer (lower-overhead jitter).
* SNI rotation, port hopping (v0.6).
