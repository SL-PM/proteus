# M2.4 Dispatch — Research Note

**Branch:** `v0.4/m1-m2-dispatch`
**Date:** 2026-05-27
**Status:** Findings only — no code change in this commit.

> The v0.4 plan-doc [§4.3 "Approach C"](PROTEUS-v0.4-plan.md) reads
> well on paper but does not survive contact with the H3 RFC. This
> document captures the discrepancy, the options we considered, and
> the recommendation for v0.4-rc.1.

---

## 1. The plan-doc design

> Drop the `proteus/0.3` ALPN. Server advertises only `h3`. Distinguish
> PROTEUS at the first-frame level (`AUTH_REQUEST` as the first frame
> on the first bidi stream = PROTEUS; H3 `SETTINGS` frame = decoy).

The assumed wire choreography:

1. Server accepts QUIC connection (ALPN-negotiated `h3`).
2. Server reads the first bidirectional stream's first frame.
3. Branch on the frame type byte — `0x00` = PROTEUS `AUTH_REQUEST`,
   `0x01` = H3 `HEADERS`, etc.

## 2. Why this doesn't work

**RFC 9114 §6.2 (HTTP/3 Streams):**

> Each side MUST initiate a single control stream at the beginning
> of the connection and send its SETTINGS frame as the first frame
> on this stream. […]
> A control stream is indicated by a **stream type of 0x00**.

Per RFC 9000, stream type bytes are only present on **unidirectional**
streams. So:

| Client → server order | Stream kind |
|---|---|
| `t = 0`     (real H3 client) | **uni** (control, sends `SETTINGS`) |
| `t = 0 + ε` (real H3 client) | bidi (the actual request, sends `HEADERS`) |
| `t = 0`     (PROTEUS client) | bidi (sends `AUTH_REQUEST`) |

The plan-doc's "first frame on the first bidi stream" inspection is
race-sensitive in the real-H3 case — the H3 control uni stream comes
*before* the bidi. The plan-doc design implicitly assumes both
protocols open a bidi first, which is wrong for H3.

## 3. Race-based discrimination has its own blocker

Naïve fix: `tokio::select!` on `accept_uni()` vs `accept_bi()`. Whichever
wins picks the protocol.

It doesn't work because **Quinn's `accept_uni()` consumes the stream**.
There's no peek API.

If `accept_uni()` wins (real H3 client), we now own the H3 control
stream containing `SETTINGS`. To then hand the connection to
`h3::server::Connection::new(conn)`, the h3 lib reads its own
connection state — and tries to accept the *next* uni stream, not
the one we already consumed. Result: h3 lib never sees `SETTINGS`,
errors out, the decoy fails.

If `accept_bi()` wins, we still have a race: an H3 client opens a
bidi for its request shortly after the uni. If that bidi arrives
before our PROTEUS client's first bidi, our `accept_bi()` could
grab the H3 request stream and try to parse it as PROTEUS frames.
The first byte would be `0x01` (H3 `HEADERS`), which is a valid
PROTEUS frame type for `AUthResponse` — confusing, not cleanly
self-disambiguating.

## 4. Considered alternatives

### Approach D1 — Manual mini-h3 server

Write a tiny H3 implementation that handles `SETTINGS` (read & discard)
and answers `HEADERS+DATA`. Skip the `h3` crate entirely.

**Cost:** ~500 LOC. Needs QPACK encoding/decoding (`qpack` crate adds
~30KB of complexity). Brittle — every H3 spec corner we get wrong is
a fingerprint.

**Verdict:** Maintainable only if v0.4 gets a dedicated multi-session
block for it.

### Approach D2 — Fork `h3` crate

Add a "delegate unknown stream type" hook to the h3 lib. PROTEUS clients
open a uni stream with a reserved-for-PROTEUS type byte; h3 lib hands
it back to us.

**Cost:** Small one-time patch to h3 (~100 LOC) but creates a fork.
Ongoing rebase burden against upstream. PRing back to h3 upstream is
possible but speculative.

**Verdict:** Doable but commits us to fork maintenance.

### Approach D3 — PROTEUS-over-H3

Every PROTEUS connection is an H3 connection. PROTEUS frames travel
inside HTTP/3 request/response bodies. Examples:

- `POST /proteus/v0.4/auth` with `AUTH_REQUEST` as the body bytes.
- WebSocket-over-h3 to `/proteus/v0.4/session` with PROTEUS frames as
  WS binary messages.
- Or: a custom HTTP method `PROTEUS-AUTH` (uncommon, may upset some
  intermediaries).

**Cost:** Major architectural change. Every PROTEUS frame gets H3
overhead (varint frame type + length). Per-stream H3 state
duplicates work the existing `proteus_core::frame` layer does.

**Verdict:** A clean v0.5 redesign. Not a v0.4-rc.1 candidate.

### Approach D4 — Race + simulated minimal H3 response

If `accept_uni()` wins, hand-roll the minimum HTTP/3 response
necessary to look like a working H3 server: skip `SETTINGS`
(non-compliant — h3 spec says server MUST send `SETTINGS` first,
but many clients tolerate its absence), open a bidi for a HEADERS
frame, write a 200 response.

**Cost:** Still needs QPACK encoding for `HEADERS`. Non-spec
compliant — clients that check `SETTINGS` reject the server. Curl
and Chrome do appear to tolerate missing `SETTINGS` in practice, but
this is testing-against-implementations rather than testing-against-spec.

**Verdict:** Brittle. The reason we want a high-fidelity decoy in
M3.4 is to look like real nginx; a non-compliant response defeats
the purpose.

### Approach D5 — Keep `proteus/0.3` ALPN, defer M1.4+M2.4

Accept that v0.4 keeps the same ALPN tell as v0.3. The rest of v0.4
(decoy fidelity, cert from CA, AEAD wrapping, migration, 0-RTT)
still closes other DPI vectors. The ALPN-drop bundles with Approach
B (cover-host upstream forwarding) in v0.4-rc.2 or later — that
work requires the same fundamental rearchitecture, so doing it
once is cleaner than doing it twice.

**Cost:** Zero code change. One plan-doc revision.

**Verdict:** Recommended for v0.4-rc.1.

## 5. Recommendation for v0.4-rc.1

**Adopt D5.** Mark M1.4 and M2.4 as deferred in the plan-doc's §9
milestones table. Document in §4.3 that the original Approach C
sketch did not survive H3-RFC contact and that Approach B / A
remains the realistic path.

This unblocks the remaining v0.4-rc.1 work:

- M5.4.1 (wire-format AEAD wrapping) — still doable, just runs on
  the `proteus/0.3` ALPN dispatch path that v0.3 already has.
- M6.4 (0-RTT resumption) — independent of ALPN choice.
- M8.4 (operator fetch-decoy utility) — independent.
- M9.4 (rc.1 sign-off) — needs to scope its capture-comparison test
  honestly: v0.4-rc.1 still has the ALPN tell.

## 6. What lands on this branch

Nothing in code. This branch carries the research note plus a
plan-doc revision marking M1.4+M2.4 deferred. Once landed (whether
merged to main or kept as a record on this branch), the v0.4-rc.1
sequence can proceed on main without ALPN-drop pressure.
