# M5.5 — v0.5-rc.1 sign-off (wire-pattern padding)

**Branch:** `v0.5/m1.5-padding-core`
**Roadmap:** [`PROTEUS-v0.5-plan.md`](PROTEUS-v0.5-plan.md) §5 acceptance criteria
**Status:** Done. Bucket-padding + idle dummy traffic shipped and
verified at the wire level. Profile-driven sampling + timing jitter
remain explicitly deferred to v0.5-rc.2 (§4 below).

---

## 1. What v0.5-rc.1 ships

Two mechanisms that reduce the v0.4 wire fingerprint:

1. **Bucket-padding (M1.5 + M2.5).** Every PROTEUS frame's on-wire
   `payload_len` is rounded up to one of `{128, 256, 512, 1024, 1500}`
   bytes. Eliminates the per-frame size leak — an observer can no
   longer read the exact length of each AUTH_REQUEST / PROXY_OPEN /
   DATA frame off the wire.
2. **Idle dummy traffic (M3.5).** The server emits a PING frame after
   a configurable quiet interval (default 5 s), padded to a
   configurable bucket (default 1024). Eliminates the "PROTEUS-idle =
   total silence" signal that distinguishes it from real H3 sessions
   (which emit periodic QUIC PINGs).

Both default OFF — a v0.4 deployment is unchanged until the operator
flips `padding.enabled` / `idle_padding.enabled`.

## 2. Acceptance criteria — checklist

From [`PROTEUS-v0.5-plan.md`](PROTEUS-v0.5-plan.md) §5:

- [x] **`proteus_core::padding` module + PADDED-flag protocol with
      round-trip tests.** 15 unit tests in `padding.rs` + 5 wire-level
      tests in `frame.rs`. (M1.5)
- [x] **Frame encode-side pads every proxy-stream frame when
      `padding.enabled`.** Server, client, and udp-test all route
      through `write_frame_*_maybe_padded`. (M2.5)
- [x] **Frame decode-side strips padding transparently.** `read_frame`
      + `read_frame_aead` auto-depad; `FLAG_PADDED` cleared before the
      frame reaches application code. (M1.5/M2.5)
- [x] **Server-side idle-padding timer emits dummy PINGs.** Bridge
      send loops `select!` on an idle deadline; receive loops skip
      inbound PINGs. (M3.5)
- [x] **Integration test: ≥95% of emitted frames land on a bucket.**
      `tests/padding.rs::padded_server_frames_land_on_buckets`
      observed 100% (4/4). (M4.5)
- [x] **Sign-off doc** (this file). (M5.5)
- [x] **No regression with `padding.enabled = false`.** All pre-v0.5
      tests pass unchanged; the padding-off write path is byte-for-byte
      identical to v0.4 (proved by
      `frame::tests::maybe_padded_with_none_matches_plain_write`).

## 3. Wire-level evidence

The M4.5 integration test reads the server→client proxy DATA frames
**at the raw QUIC-stream level** (16-byte header parse, no AEAD-open),
so the numbers below are the actual on-wire `payload_len` values.

### 3.1 Padded (padding.enabled = true, default buckets)

| Echo payload | Observed wire `payload_len` | Bucket |
|---:|---:|---:|
| 5 B    | 128  | ✅ 128 |
| 100 B  | 128  | ✅ 128 |
| 300 B  | 512  | ✅ 512 |
| 1000 B | 1024 | ✅ 1024 |

100% of observed frames (4/4) landed on a configured bucket. The
real payload size is no longer recoverable from the wire length — a
5-byte and a 100-byte response are indistinguishable (both 128).

### 3.2 Unpadded control (padding.enabled = false)

A 300-byte echo appears on the wire as **~316 bytes** (300 payload +
16-byte ChaCha20-Poly1305 tag) — NOT a bucket value. This control
case (`unpadded_server_frames_are_not_bucket_aligned`) confirms the
padded result is a real wire change, not a measurement artifact.

### 3.3 Idle dummy traffic

Verified by live smoke (server `idle_padding.interval_secs = 1`,
`bucket = 512`): a deliberately slow HTTP endpoint (3 s server-side
delay = 3 idle cycles) round-tripped through the SOCKS5 proxy and
returned its body intact. This proves idle PINGs are (a) emitted
during quiet periods, (b) skipped by the receiving client, and
(c) do not desync the per-stream AEAD counters or disrupt the real
response.

## 4. What v0.5-rc.1 does NOT achieve (honest limitations)

Bucket-padding is a *quantizer*, not a *mimic*. The residual gaps:

* **Distribution shape still differs from real H3.** A real cover
  host produces a smooth, long-tailed packet-size distribution.
  PROTEUS-v0.5-rc.1 produces a 5-spike distribution (one spike per
  bucket). A statistical attacker (A7) who models the size histogram
  can still distinguish the two. **Closing this needs profile-driven
  sampling** — drawing frame sizes from a recorded cover-host
  distribution ([`fingerprint-profile.example.yaml`](fingerprint-profile.example.yaml)
  is the ready schema). Deferred to **v0.5-rc.2**.
* **Timing is unchanged.** No inter-arrival jitter. The *timing*
  fingerprint of PROTEUS request/response patterns is untouched.
  Deferred to **v0.5-rc.2**.
* **`proteus/0.3` ALPN still distinctive.** Orthogonal to padding;
  re-scoped to v1.0 (see
  [`m2.4-dispatch-research.md`](m2.4-dispatch-research.md)).
* **Idle PINGs are a fixed-rate, fixed-size pattern.** A constant
  512-byte frame exactly every 5 s is itself a (weak) signal. Real
  H3 PING cadence is irregular. Profile-driven idle timing is
  rc.2 work.
* **No pcap-level cross-check in CI.** The wire evidence in §3 comes
  from in-process raw-frame inspection, which sees PROTEUS frame
  sizes directly. A tcpdump/tshark capture would show QUIC *packet*
  sizes (PROTEUS frame + QUIC framing + UDP/IP), one layer out. The
  existing `scripts/capture-*.sh` (M15) can produce that manually;
  it's not wired into the automated suite because it needs root.

**Net:** v0.5-rc.1 closes the *per-frame size* leak (A5 partial) and
the *idle-silence* leak. It does **not** claim to defeat a
statistical traffic-analysis attacker (A7) — that is rc.2's bar.

## 5. Threat-model delta

| Adversary | v0.4 | v0.5-rc.1 |
|---|---|---|
| A5 active DPI / wire-pattern | partial (AEAD inside tunnel; per-frame sizes leak) | **mostly closed** — per-frame size quantized; idle filled |
| A7 statistical analysis | ✗ | **partial** — coarse quantization only; distribution shape + timing remain |
| A6, A8–A11 | unchanged | unchanged |
