# M10.5 — token-bucket pacer sign-off

**Branch:** `v0.5/m9.5-pacer`
**Roadmap:** [`PROTEUS-v0.5-plan.md`](PROTEUS-v0.5-plan.md) §11 (jitter) — pacer refinement
**Status:** Done. Token-bucket burst allowance shipped on top of the
rc.2 per-frame jitter, lowering its latency cost for bursty /
interactive traffic.

---

## 1. What this ships

A **token-bucket burst allowance** for the send-path timing jitter.
Config gains `timing_jitter.burst` (default 0):

* `burst = 0` → unchanged rc.2 behavior: every frame pays the full
  jittered delay.
* `burst = N` → the first `N` queued frames send with **zero added
  delay**; sustained over-rate traffic is then paced at ~one frame per
  `mean(min,max)` with a jittered wait. The allowance **refills over
  idle time**, so a quiet gap restores the free burst — matching real
  interactive patterns (click → small burst → idle → click).

Implemented as `proteus_core::jitter::Pacer`, a stateful token bucket
with an injectable clock. A neat property keeps it backward-compatible:
**capacity 0 reduces to exactly the rc.2 per-frame jitter** (the bucket
can never hold a token, so every frame takes the wait branch). So the
default changes nothing and all rc.2 tests pass unchanged.

## 2. Honest correction to the rc.2 sign-off

[`m8.5-timing-jitter-signoff.md`](m8.5-timing-jitter-signoff.md) §4
floated "a token-bucket pacer … would cost less." That is true **only
for bursty / interactive traffic**. It does **NOT** raise the
*sustained* bulk-throughput ceiling:

> Any time-spacing scheme rate-limits sustained output to
> ~`frame_size / mean_interval`. A token bucket lets short bursts (≤
> capacity) through instantly, but once the bucket drains, sustained
> traffic is paced at the refill rate — the same ceiling as per-frame
> jitter.

What the pacer genuinely buys:

* **Interactive latency**: a request + a few response frames that fit
  in the burst incur *zero* added delay, vs. per-frame jitter delaying
  every one. This is the case where jitter's latency hurt most and
  where the timing fingerprint matters most.
* **Operator knob**: size `burst` to the typical burst of your
  workload; bulk users still disable jitter or accept the rate cap.

The `sustained_rate_is_bounded_by_mean` unit test pins the ceiling
explicitly (only `capacity` free sends at a fixed instant).

## 3. Acceptance criteria — checklist

- [x] **`Pacer` token bucket with time-refill, deterministic.** 5 unit
      tests (`crates/proteus-core/src/jitter.rs`): burst frames free,
      capacity-0 == per-frame jitter, idle refill restores the burst,
      zero-range no-op, sustained rate bounded by capacity. Injectable
      clock + RNG → no wall-clock flakiness. (M9.5)
- [x] **`burst` config field, default 0, backward-compatible.** rc.2
      semantics preserved when unset. (M9.5)
- [x] **Wired into both bridges.** `bridge_quic_tcp` + `bridge_quic_udp`
      build a per-stream `Pacer` and pace the DATA send path; idle
      PINGs still excluded. Threaded through server (bind → … → bridge)
      and client. `JitterPlan { jitter, burst }` replaces the bare
      `Jitter` on the bridge boundary. (M10.5)
- [x] **Integration test: burst bypasses pacing.**
      `tests/jitter.rs::token_bucket_burst_does_not_serialize` — with a
      100ms per-frame delay but `burst = 50`, the echo frame rides a
      free token and returns in < 80ms (would be ~100ms+ per frame
      under per-frame jitter), byte-identical. (M10.5)
- [x] **No regression.** All prior tests pass; capacity-0 path is the
      rc.2 behavior. 159 tests total (+6 since v0.5.0: 5 Pacer units + 1
      burst integration test).

## 4. Evidence

* Unit (deterministic, injectable clock): `cargo test -p proteus-core jitter`
  — 10 tests (5 Jitter + 5 Pacer).
* Integration: `cargo test -p proteus-server --test jitter` — 3 tests
  (per-frame correctness/timing from rc.2, multi-frame ordering, and
  the new burst-bypass timing test).

## 5. Threat-model delta

**None.** The pacer is an operability/latency refinement of the rc.2
jitter, not a new anti-fingerprinting mechanism. A7 status is unchanged
from v0.5.0: send timing is *decorrelated*, not *matched*. If anything,
a large burst means short flows are **not** timing-shaped at all (they
pass instantly) — operators trading latency for obfuscation should size
`burst` small. Documented in CONFIG.md.

## 6. Deferred (unchanged)

* Profile-driven size + inter-arrival sampling (needs a capture corpus)
  — the real A7 closer.
* Coalescing small frames to raise the *bulk* ceiling.
* SNI rotation, port hopping (v0.6).

## 7. Versioning

Backward-compatible feature addition, default off → **v0.5.1**.
