# M16.5 — profile-driven padding wired end-to-end

**Branch:** `v0.5/m15.5-profile-core` (M15.5) → `v0.5/m16.5-profile-wire`
**Roadmap:** [`PROTEUS-v0.5-plan.md`](PROTEUS-v0.5-plan.md) §6, building on the
M14.5 feasibility result.
**Status:** Done for the *pad-up* case. Fragmentation (sizing a frame
*below* its payload) is deferred to its own milestone — see §4.

---

## 1. What this ships

The first wire-shaping mechanism that *matches* a target distribution
rather than merely decorrelating. Instead of rounding each frame up to
a fixed bucket, the sender draws a target wire `payload_len` from a
configured size profile and pads up to it, so the emitted size
distribution follows the profile.

* **M15.5 (core):** `fingerprint::Distribution::sample_ge(min, rng)` —
  inverse-CDF sampling conditioned on key ≥ `min` (a frame can only be
  padded *up*, so the target must be ≥ payload + trailer). Plus
  `ProfilePaddingConfig { enabled, sizes: [{size, weight}] }` with
  `to_distribution()` / `candidate_sizes()` / `validate()`.
* **M16.5 (wire-up):** both bridges (`bridge_quic_tcp` /
  `bridge_quic_udp`) take an `Option<Distribution>` profile. Per DATA
  frame they sample one target ≥ payload+trailer and pad to exactly it
  (reusing the existing `pick_bucket`/`pad_frame` path via a
  single-element bucket). Threaded through server + client; idle PINGs
  excluded. `profile_padding` takes precedence over bucket `padding`;
  its candidate sizes double as the read-buffer hint and the fallback.

Default off — `profile_padding.enabled: false` is byte-for-byte v0.5.1.

- [x] **`sample_ge` conditional sampler, deterministic.** 4 unit tests
      (respects bound, None when nothing fits, min=0 == full sample,
      reproduces the conditional distribution). (M15.5)
- [x] **`ProfilePaddingConfig` + validation.** 3 unit tests
      (default off, weight expansion, enabled-but-weightless rejected).
      Added to ServerConfig + ClientConfig, default off. (M15.5)
- [x] **Wired into both bridges + threaded server/client.** RNG created
      and dropped within the sync sizing statement so the bridge future
      stays `Send`. (M16.5)
- [x] **End-to-end: live frames carry profile sizes, not buckets.**
      `tests/fingerprint.rs::profile_padding_sizes_live_frames_from_the_profile`
      — a profiled server (`{200,440,900}`, disjoint from the default
      buckets) sending 50-byte echoes produces observed wire sizes
      **exactly `{200, 440, 900}`** and never a `{128,256,512,1024,1500}`
      bucket. (M16.5)
- [x] **End-to-end: AEAD round-trip intact.**
      `profile_padded_data_round_trips` — a 600-byte payload reassembles
      byte-identical through the sample-then-pad path. (M16.5)
- [x] **No regression.** 182 tests pass; profile-off path unchanged.

## 3. Evidence

```
--- M16.5 live profile-padding wire sizes: {200, 440, 900} ---
```

A 50-byte payload under *bucket* padding would be 128. Under the
profile it is sampled to one of `{200, 440, 900}` — observed on the raw
QUIC wire, proving profile sizing is the active mechanism end-to-end,
not bucketing. The companion test confirms a 600-byte payload still
round-trips byte-identical through the AEAD layer.

This is the live counterpart to the M14.5 offline result
(`TV(bucketing, cover)=1.000` vs `TV(profile-sampling, cover)=0.005`):
profile padding moves the emitted size distribution onto the profile.

## 4. The honest limit — pad-up only (fragmentation deferred)

Padding can only grow a frame, never shrink it. So the target must be
≥ `payload + trailer`, and `sample_ge` restricts to profile sizes that
fit. Consequences:

* **Small / interactive payloads** (the common case) fit under most or
  all profile sizes → the emitted distribution closely follows the
  profile. This is where the size fingerprint matters most.
* **Payloads larger than every profile size** can't be sized down to a
  small profile bin. Today they fall back to bucket padding (the
  profile's candidate sizes serve as the fallback bucket set). So a
  bulk transfer's large frames are *not* yet profile-shaped.

Closing that needs **fragmentation**: split a large payload into
several smaller frames so each can hit a small profile size. That's a
real design effort (reassembly, per-fragment AEAD counters, overhead
accounting) and is its own deferred milestone (M17.5+). Until then,
profile padding is honest for interactive workloads and degrades to
bucketing for bulk.

## 5. Threat-model delta

| | |
|---|---|
| Before (v0.5.1) | size axis = bucket quantizer (5 spikes); the M12.5/M14.5 limit was "across-bucket activities stay distinguishable; bucketing ≠ a real cover distribution" |
| After (M16.5) | for payloads that fit a profile size, the emitted size distribution **matches the profile** — the first mechanism that addresses the "5-spike ≠ cover" limit, not just decorrelates |

**Caveats that remain (honest):**
* Only as good as the profile. A *synthetic* profile proves the
  mechanism; meaningful tarnung needs a profile derived from a real
  capture corpus (`scripts/capture-*.sh`; the
  [`fingerprint-profile.example.yaml`](fingerprint-profile.example.yaml)
  schema is the eventual home for that).
* Pad-up-only bias (§4): the realized distribution is the profile
  *conditioned on ≥ payload*, so a workload of large payloads can't
  reproduce a profile's small-size mass.
* Timing axis unchanged — this is the size axis only.

## 6. Versioning

Feature, default off, backward-compatible. Ships in the v0.5.x line; a
clean tag (v0.5.3 or v0.6.0) is best taken once fragmentation rounds
out the mechanism, or as an explicit "interactive-only profile padding"
increment if tagged sooner.
